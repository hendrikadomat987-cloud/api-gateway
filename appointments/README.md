# Appointments Service

Multi-tenant CRUD service for managing customer appointments with scheduling, duration, and status tracking.

---

## Architecture

```
Client → API Gateway (JWT auth + tenant injection) → n8n Webhook → PostgreSQL RPC (RLS)
```

---

## Data Model

| Field            | Type        | Required | Notes                                                        |
|-----------------|-------------|----------|--------------------------------------------------------------|
| id              | UUID        | auto     | Generated by DB                                              |
| tenant_id       | UUID        | JWT only | Never from client — injected by gateway                      |
| customer_id     | UUID        | yes      | Must be a valid UUID                                         |
| scheduled_at    | timestamptz | yes      | ISO 8601 datetime                                            |
| duration_minutes| integer     | no       | 1–1440 min, default 60                                       |
| status          | text        | no       | `scheduled` (default) \| `confirmed` \| `cancelled` \| `completed` |
| notes           | text        | no       | Free-form text                                               |
| created_at      | timestamptz | auto     | Set on insert                                                |
| updated_at      | timestamptz | auto     | Updated on every change                                      |

## API Endpoints

| Method | Path                        | Operation              |
|--------|------------------------------|------------------------|
| POST   | /api/v1/appointments        | Create appointment     |
| GET    | /api/v1/appointments        | List all appointments  |
| GET    | /api/v1/appointments/:id    | Get appointment by ID  |
| PUT    | /api/v1/appointments/:id    | Partial update         |
| DELETE | /api/v1/appointments/:id    | Delete (idempotent)    |

### Request / Response Examples

**POST /api/v1/appointments**
```json
// Request body
{
  "customer_id":      "uuid",
  "scheduled_at":     "2026-04-01T10:00:00Z",
  "duration_minutes": 30,
  "status":           "scheduled",
  "notes":            "Initial consultation"
}

// Response 200
{
  "success": true,
  "data": { "id": "uuid", "tenant_id": "uuid" }
}
```

**GET /api/v1/appointments/:id**
```json
// Response 200
{
  "success": true,
  "data": {
    "id": "uuid",
    "tenant_id": "uuid",
    "customer_id": "uuid",
    "scheduled_at": "2026-04-01T10:00:00Z",
    "duration_minutes": 30,
    "status": "scheduled",
    "notes": "...",
    "created_at": "2026-03-26T00:00:00Z",
    "updated_at": "2026-03-26T00:00:00Z"
  }
}
```

**PUT /api/v1/appointments/:id** — at least one field required
```json
{ "status": "confirmed" }
```

**Error format**
```json
{ "success": false, "error": { "code": "VALIDATION_ERROR", "message": "..." } }
```

## Security

- `tenant_id` is **never** accepted from the client body
- Gateway extracts `organization_id` (or `tenant_id`) from the verified JWT and injects it before forwarding
- PostgreSQL RLS + FORCE RLS enforces tenant isolation at the DB layer
- All IDs validated as UUIDs before forwarding to n8n
- `duration_minutes` validated in range 1–1440
- `scheduled_at` validated as a parseable ISO 8601 timestamp

2. Mount `tenantInjectionMiddleware` before forwarding

## Deployment Phases

### Phase 1 — Generate
All artifacts in this folder are generated and ready.

### Phase 2 — Apply / Deploy

Execute in this exact order:

1. **Apply SQL to Supabase**
   ```sql
   -- Run appointments.sql in Supabase SQL Editor or via psql
   \i appointments.sql
   ```

2. **Import and activate n8n workflows**
   - Open n8n → Workflows → Import from file
   - Import each `_appointments_*.json` file
   - Activate all 5 workflows
   - Verify webhook URLs are live

3. **Deploy gateway changes**
   - Add `appointments: 'appointments_service'` to `api-gateway/config.js`
   - Mount the router from `appointments.gateway.patch.js`
   - Restart / redeploy the gateway

### Phase 3 — Test

> **Run tests ONLY after all three Phase 2 steps are complete.**

```bash
# From project root
node test-engine/run-tests.js appointments/appointments.crud.test.js
node test-engine/run-tests.js appointments/appointments.gateway-security.test.js
node test-engine/run-tests.js appointments/appointments.rls.test.js
```

Prerequisites before running:
- `appointments.sql` applied to Supabase ✓
- All 5 n8n workflows imported and active ✓
- Gateway restarted with appointments service registered ✓
- `VALID_CUSTOMER_ID` in test files updated to a real customer UUID for your test tenant ✓
